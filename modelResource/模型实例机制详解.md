# 模型实例（ModelCaseReference）机制详解

## 一、什么是模型实例？

**模型实例（ModelCaseReference，简称MCR）** 是这个系统的核心概念。它不仅仅是"保存的模型"，而是一个包含了完整运行状态和数据的**任务容器**。

### 通俗比喻
想象你要做一道菜（计算任务）：
- **模型脚本** = 菜谱（固定的计算逻辑）
- **模型实例** = 一次具体的烹饪过程（包含：用的食材、做到哪一步了、最终成品）

同一个菜谱（模型）可以做很多次菜（多个实例），每次用的食材可能不同（不同的输入参数），每道菜都有独立的进度和结果。

---

## 二、模型实例包含什么？

每个模型实例都会创建一个独立的文件夹，结构如下：

```
case/
└── {模型实例ID}/                    # 以MD5哈希命名的唯一ID
    ├── lock                         # 文件锁（防止并发冲突）
    ├── identity.json                # 身份信息
    │   ├── model: 模型名称
    │   ├── model-path: 模型脚本路径
    │   ├── url: API地址
    │   ├── json: 请求参数（输入数据）
    │   └── dependencies: 依赖的其他实例ID列表
    ├── time/                        # 时间戳记录
    │   └── {时间戳}                 # 用于缓存管理和清理
    ├── status/                      # 运行状态
    │   └── {状态码}                 # 1=未锁定, 2=锁定, 4=运行中, 8=完成, 32=错误
    ├── response.json                # 响应结果（输出数据）
    └── result/                      # 计算产生的文件
        └── 各种输出文件...
```

## 三、为什么需要模型实例？

### 3.1 四大核心作用

```
┌─────────────────────────────────────────────────────────────────┐
│                     模型实例的核心价值                            │
├───────────────┬─────────────────────────────────────────────────┤
│ 1. 状态管理    │ 追踪每个任务的执行状态（等待/运行中/完成/错误）    │
├───────────────┼─────────────────────────────────────────────────┤
│ 2. 数据持久化  │ 保存输入参数、中间结果、最终输出                   │
├───────────────┼─────────────────────────────────────────────────┤
│ 3. 依赖管理    │ 记录任务间的依赖关系，自动调度执行顺序             │
├───────────────┼─────────────────────────────────────────────────┤
│ 4. 缓存复用    │ 相同输入直接返回已有结果，避免重复计算             │
└───────────────┴─────────────────────────────────────────────────┘
```

### 3.2 实例 vs 脚本的区别

| 特性 | 模型脚本 (.py文件) | 模型实例 (MCR) |
|------|-------------------|---------------|
| **性质** | 静态的代码 | 动态的任务 |
| **数量** | 每个模型1个 | 可以无限多个 |
| **内容** | 计算逻辑 | 输入数据 + 状态 + 结果 |
| **复用** | 所有实例共享 | 每个实例独立 |
| **生命周期** | 永久存在 | 创建 → 运行 → 完成/保存 |

## 四、模型执行流程详解

### 4.1 完整执行流程

```
用户请求
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│ 1. PARSING 阶段（解析）                                            │
│    - 检查是否有相同参数的已完成实例（缓存命中直接返回）               │
│    - 没有缓存则创建新的模型实例                                      │
│    - 解析依赖关系，生成依赖实例列表                                   │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│ 2. 依赖检查阶段                                                    │
│    - 检查所有前置依赖实例是否已完成                                   │
│    - 如果有依赖出错，标记当前实例为错误                               │
│    - 如果依赖未完成，加入等待队列                                     │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│ 3. 队列调度阶段（后台守护进程）                                      │
│    - 检查并发数限制（最多20个同时运行）                              │
│    - 依赖全部就绪后，启动子进程运行模型                               │
│    - 更新状态为 RUNNING                                            │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────────┐
│ 4. 模型运行阶段（子进程中执行）                                      │
│    - 加载模型脚本（.pyc字节码）                                      │
│    - 执行 run_xxx_mcr() 函数                                        │
│    - 计算完成后，调用 make_response() 保存结果                        │
│    - 更新状态为 COMPLETE                                           │
└──────────────────────────────────────────────────────────────────┘
    │
    ▼
返回实例ID给用户
用户可通过API查询状态和获取结果
```

### 4.2 流程图示例

以 **Risk Level（综合风险评估）** 为例：

```
用户调用 Risk Level
    │
    ├───► 创建 Soil Composition 实例 (Dsed) ─────┐
    ├───► 创建 Slope Protection 实例 (PL) ───────┤
    ├───► 创建 Load Control 实例 (LC) ───────────┤
    ├───► 创建 Height Difference 实例 (Zb) ──────┤───► 依赖 Section View
    ├───► 创建 Slope Rate 实例 (Sa) ─────────────┤───► 依赖 Section View
    ├───► 创建 Nearshore Flush 实例 (Ln) ────────┤───► 依赖 Region Flush → Section View
    ├───► 创建 Flow Equivalent 实例 (PQ) ────────┤
    ├───► 创建 Anti-Impact Speed 实例 (Ky) ──────┤───► 依赖 Section View + Hydrodynamic
    ├───► 创建 Water-Level Fluctuation 实例 (Zd)─┤───► 依赖 Section View + Hydrodynamic
    │
    └───► 创建 Risk Level 实例（依赖上述9个）

所有实例创建完成后，后台守护进程开始调度：

批次1: Soil, Protection, Load, Flow Equivalent （独立，立即执行）
批次2: Section View, Region Flush （基础依赖）
批次3: Height Difference, Slope Rate （依赖Section View）
批次4: Nearshore Flush （依赖Region Flush完成后的Section View）
批次5: Anti-Impact Speed, Water-Level Fluctuation （依赖Section View + Hydrodynamic）
批次6: Risk Level （等待所有前9个完成）
```

## 五、代码层面的工作机制

### 5.1 创建实例的核心代码

```python
# 在模型脚本中（如 soilComposition.py）
def PARSING(self, request_json: dict, model_path: str, other_dependent_ids: list[str]=[]):
    # 创建模型实例，但不立即执行
    soil_composition_mcr = model.ModelCaseReference.create(
        config.API_MI_SOIL_COMPOSITION,  # API地址
        request_json,                     # 输入参数
        NAME,                             # 模型名称
        model_path,                       # 脚本路径
        other_dependent_ids               # 额外依赖的实例ID
    )
    
    # 返回实例列表，最后一个是要执行的核心实例
    return [soil_composition_mcr]
```

### 5.2 启动执行的核心代码

```python
# 在 modelLauncher.py 中
def run(self, request_json: dict, other_dependent_ids: list[str] = []) -> MCR:
    # 1. 解析依赖，创建所有需要的实例
    default_pre_mcrs = self.parse(request_json, self.program_path, other_dependent_ids)
    mcrs = other_pre_mcrs + default_pre_mcrs
    
    # 2. 检查是否可以运行
    if mcr_checker(mcrs):
        # 3. 生成命令行参数：[pyc文件路径, 依赖实例ID1, 依赖实例ID2, ..., 核心实例ID]
        command = [self.program_path] + [mcr.id for mcr in mcrs]
        
        # 4. 放入进程队列，由守护进程调度执行
        Process_Queue().put(command)
    
    # 5. 立即返回最后一个实例（核心实例）给用户
    return mcrs[-1]
```

### 5.3 子进程中如何运行

```python
# 当守护进程调度到该任务时，会执行：
python {pyc文件路径} {依赖ID1} {依赖ID2} ... {核心实例ID}

# 在模型脚本的 __main__ 块中：
if __name__ == '__main__':
    # 从命令行参数获取实例ID
    v1 = sys.argv[1]    # 第一个依赖实例ID
    v2 = sys.argv[2]    # 第二个依赖实例ID
    ...
    vn = sys.argv[n]    # 核心实例ID
    
    # 通过ID重新打开实例对象
    section_view_mcr = model.ModelCaseReference.open_case(v1)
    hydrodynamic_mcr = model.ModelCaseReference.open_case(v2)
    core_mcr = model.ModelCaseReference.open_case(vn)
    
    # 执行核心计算函数
    run_anti_impact_speed_mcr(core_mcr)
```

## 六、缓存机制（重要！）

### 6.1 缓存原理

```python
# ModelCaseReference 的 __init__ 方法中：
self.id = util.generate_md5(request_url + json.dumps(request_json))
```

**实例ID = MD5(API地址 + 输入参数JSON)**

这意味着：
- 相同的API + 相同的参数 = 相同的ID
- 如果该ID的实例已经存在且状态为COMPLETE，直接返回已有结果
- 不会重复计算，大大提高效率

### 6.2 缓存示例

```python
# 第一次调用
request = {"hs": 0.5, "hc": 2.0}
mcr1 = launcher.run(request)  # 创建新实例，执行计算

# 第二次调用（相同参数）
mcr2 = launcher.run(request)  # 发现已有完成实例，直接返回mcr1的ID

# mcr1.id == mcr2.id，没有重复计算
```

## 八、回答问题

### Q1: 模型实例是干什么用的？

**答：** 模型实例是这个系统的"任务管理单元"。它实现了：
1. **异步执行** - 提交任务后立刻返回，后台慢慢算
2. **进度追踪** - 随时查询任务进行到哪一步了
3. **结果缓存** - 相同输入直接返回已有结果，省时间
4. **依赖管理** - 自动处理复杂的模型依赖关系
5. **错误隔离** - 某个模型出错不影响其他任务

### Q2: 为什么需要模型实例？

**答：** 没有这个机制的话：
- ❌ 无法知道任务何时完成
- ❌ 无法处理模型间的依赖关系
- ❌ 每次调用都要重复计算，效率低下
- ❌ 出错后无法追踪和恢复

有了实例机制：
- ✅ 提交任务 → 返回ID → 轮询状态 → 获取结果
- ✅ 复杂依赖自动解析和调度
- ✅ 相同参数秒级返回（缓存）
- ✅ 完整的状态和日志记录

### Q3: 九个模型都会被保存为实例吗？

**答：** 是的！当调用 **Risk Level** 时：

1. 系统会创建 **10个模型实例**（9个子模型 + 1个综合模型）
2. 每个实例都有独立的ID、状态、结果
3. 9个子模型先并行/串行执行
4. 全部完成后，Risk Level 实例汇总结果
5. 所有实例都会被持久化保存

这样做的好处：
- 可以单独查询每个子模型的结果
- 可以复用子模型的结果（例如两个不同的Risk Level计算共享同一个Soil Composition结果）
- 便于调试和追踪问题

## 九、实际使用示例

### 9.1 查看实例状态

```python
from model.modelCaseReference import ModelCaseReference as MCR

# 通过ID获取实例状态
status = MCR.get_case_status(case_id)
# 返回值：1=未锁定, 2=锁定, 4=运行中, 8=完成, 32=错误

# 获取更友好的状态描述
status_text = MCR.check_case_status(case_id)
# 返回值：'UNLOCK'/'LOCK'/'RUNNING'/'COMPLETE'/'ERROR'/'NONE'
```

### 9.2 获取实例结果

```python
# 获取响应数据（包含计算结果）
response = MCR.get_case_response(case_id)
print(response)
# {
#   "case-id": "a1b2c3d4...",
#   "model": "Soil Composition",
#   "Dsed": 0.25,
#   "risk-level": [0,0,1,0]
# }
```

### 9.3 删除实例

```python
# 删除实例及其所有数据
success = MCR.delete_case(case_id)
```

## 十、总结

模型实例（MCR）是这个系统的**核心设计**，它将：

```
静态的模型脚本 ────────┐
                    ├───► 动态的、可管理的、可追踪的、可缓存的任务单元
动态的输入参数 ────────┘

核心价值：
1. 异步化 - 提交即返回，后台执行
2. 可追踪 - 随时查看进度和状态  
3. 可复用 - 智能缓存避免重复计算
4. 可管理 - 依赖自动解析和调度
5. 可维护 - 完整日志便于调试
```

这就是为什么你看到的模型代码中，总是要返回 `ModelCaseReference` 对象，因为它是连接用户请求和实际计算的桥梁，是整个系统能够高效、可靠运行的关键！
